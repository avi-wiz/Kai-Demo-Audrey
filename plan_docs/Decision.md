# Decision.md — Kai Open-Ended Planner

**Author:** Avi (PM, Kai) + Claude
**Date:** 2026-05-21
**Branch:** `ui-overhaul`
**Status:** Design locked for v1. v2 / v3 scoped at high level.

---

## 1. Problem Statement

Today's pipeline is **keyword → frozen fixture**. The fixture fuses three things — intent, data slice, and presentation — into a single JSON blob. When a client asks an *adjacent* question ("show overdue tasks but for Marcus", "Magnolia's revenue this quarter vs last") the system has no path: every wired capability is hardcoded, and the LLM classifier only force-fits queries into the nearest fixture.

**Goal:** Kai should answer any **adjacent** question — anything built on the entities, widgets, and capabilities already wired — without hardcoded fixtures. Scripted demo flows remain rock-solid; improvised queries get a planned response composed at runtime.

**Non-goal (v1):** Arbitrary open-ended questions outside the Audrey data model.

---

## 2. Architectural Direction

Decouple the three things fixtures currently fuse:

1. **Intent** — what is the user asking for?
2. **Data** — which slice of Audrey data answers it?
3. **Presentation** — which widget(s) render that data well?

The LLM's job becomes **planning** (pick widgets + pick data queries + bind them), not generating UI from scratch. Existing fixtures become (a) the fast path for canonical demo flows and (b) exemplars the planner learns from.

**Three new layers, in build order:**

1. **Queryable data layer** — typed query API over `AUDREY_*` arrays. Exposed to the planner as tools.
2. **Widget catalog** — machine-readable spec per registered widget (`dataShape`, `goodFor`, `example`). The planner picks from this catalog.
3. **Planner / composer turn** — LLM produces a small **Plan IR**, which we compile into Frame JSON and run through the existing FrameParser pipeline. **No new rendering engine.**

---

## 3. The v1/v2/v3 Roadmap

### v1 — Adjacent reads, single-turn, static reasoning card

- Read-only planner. No writes.
- IR-based plan output, validated with zod, compiled to Frame JSON.
- Fast-path keyword routes win over the planner (scripted demo flows stay deterministic).
- Confidence-gated LLM classifier + `matchQuery`: low-confidence → planner.
- SKU-lookup guard hands off to planner with a hint instead of terminating.
- **6 wide query tools** (`queryCustomers`, `queryLeads`, `queryOrders`, `queryProducts`, `queryTasks`, `queryReps`) with rich filter args + embedded aggregations + auto-hydrated foreign keys.
- Max **4 widgets** per planner response.
- UW-014 AgentReasoningCard streams live during plan execution, collapses to a "Thought for Ns" summary on completion (mimics Claude.ai).
- Closing text generated **after** all widgets hydrate. Never empty — generates *insights* on empty result sets (not blank text).
- Fallback card on planner failure: honest message + 2-3 chips toward closest wired capability.
- Generic chips below planner responses ("Save to artifacts", "Show me more").
- Kill switch: `KAI_PLANNER_DISABLED` env / settings flag.
- Logging: structured console + `window.lastPlannerRun` debug payload.
- Models: **Opus 4.6 planner**, **Sonnet 4.6 closing text + data hydration**.

### v2 — Charts, dynamic chips, conversation history, progressive UX

- **Chart generation:** planner picks CH-001 LineChart, chooses axes + series.
- **Dynamic action chips:** planner generates context-specific chips per response.
- **Conversation history visible to planner** — supports "now group by month"-style refinement.
- **Progressive widget skeletons** stream as the plan hydrates (replacing v1's static reasoning card as the loading affordance — reasoning card still collapses to summary at the end).
- **Hybrid mode:** planner can augment a wired-cap output (e.g., Cap 7 pipeline report → "now filter to Marcus").
- **Widget-level graceful degradation:** if one widget's data shape fails validation, drop that widget and render the rest (v1 fails the whole plan).
- **Persistent planner logging** to database; Streamlit dashboard hooks up downstream.

### v3 — Writes, LLM-chained joins, multi-turn refinement, cross-page

- **Read+write planner:** can call `addTask`, `addLead`, `addCustomer` mutators with explicit confirmation UX.
- **LLM-chained joins:** for queries that auto-hydrate can't cleanly express, planner chains query tools (v1/v2 keep auto-hydration on read).
- **Many narrow tools** considered (if wide tools hit expressiveness limits).
- **Multi-turn refinement:** full conversational state, not just last-frame visibility.
- **Cross-page navigation suggestions** generated by planner.
- **Hybrid mode** matures: planner-augmented wired-cap outputs become a first-class flow.

## 4. Decisions Locked (v1)

### 4.1 Scope & UX

| # | Decision | Rationale |
|---|---|---|
| 1 | **Scope = adjacent questions only.** | 80% of "client improv" outcome at ~20% of effort. Arbitrary open-ended is v2+. |
| 2 | **Keyword fast-paths win over planner.** Planner only fires when keyword routes don't match AND confidence-gated LLM stages defer. | Protects scripted demo flows. |
| 3 | **Latency budget:** stream the plan, render widgets as they hydrate. | UX > raw speed. |
| 4 | **Failure modes:** (a) fallback "I can't answer that" card with chips, **and** (c) degrade to text-only LLM answer. Never (b) live-retry visible to user. | Predictable behavior on demo day. |
| 5 | **New route** `/api/kai/plan`. | Keeps existing `/api/kai/generate` pipeline untouched. |
| 6 | **Closing text renders only after all preceding widgets are hydrated and rendered.** Never empty — generates insights from self-explanatory data. | Avoids half-painted responses. |
| 7 | **Reasoning card behavior:** UW-014 streams live ("Querying orders for Marcus → Found 12 → Rendering table"), collapses to compact "Thought for Ns" summary on completion. | Feature, not loading state. |
| 8 | **Fallback card copy:** brief honest message + 2-3 chips toward closest wired capability. | "Want me to show the Pipeline report instead?" |

### 4.2 Data Layer

| # | Decision | Rationale |
|---|---|---|
| 9 | **Reads only for v1.** Writes (addTask/addLead) reserved for v3. | Writes have Shared-context side effects + need confirmation UX. |
| 10 | **Never put raw data in the prompt.** LLM sees schemas + tool signatures + count summaries only. Must call a tool to see rows. | Prevents pattern-matching shortcuts; forces correct grounding. |
| 11 | **Few wide tools** (6 total): `queryCustomers`, `queryLeads`, `queryOrders`, `queryProducts`, `queryTasks`, `queryReps`. Rich filter args. | Adjacent queries combine 2-3 dimensions; narrow tools can't express that. |
| 12 | **Aggregations embedded in query tools** via `groupBy` / `aggregate` params. Result is a discriminated union: `{ kind: 'rows', rows } \| { kind: 'groups', groups }`. | Fewer round-trips; less chaining for the LLM. |
| 13 | **Joins: auto-hydrate on read.** Each `Order` returned already has `.customer`, `.rep` hydrated. | 35 orders — perf irrelevant. LLM chaining is where fumbles happen. |
| 14 | **Result size cap: 50 rows.** Tools return `{ truncated: true, total: N }` past that. | Forces the LLM to add filters instead of dumping. |

### 4.3 Planner IR

| # | Decision | Rationale |
|---|---|---|
| 15 | **IR-based plan output**, not direct Frame JSON. Compiled to Frame JSON by us. | Easier validation, easier streaming, easier debug/replay. |
| 16 | **IR schema:** see §5 below. | Locked. |
| 17 | **Max 4 widgets per planner response.** | Prevents kitchen-sink responses. |
| 18 | **Validation:** zod-validate IR → on failure, one silent pre-render retry with error appended → if still bad, fallback card. Never user-visible retry. | Silent retry is invisible; live-retry erodes trust. |
| 19 | **Widget data validation at compile time:** if any widget's hydrated data fails its `dataShape`, **fail the whole plan** and fall back. | Partial responses look broken. v2 relaxes to per-widget drop. |
| 20 | **Empty-result branches flagged in IR** so closing text generator produces *insight-style* copy ("No overdue tasks for Marcus this week — his pipeline is current"), not blank. | Aligned with #6. |
| 21 | **closingTextHint: null** in IR means skip the closing pass entirely. | Saves an LLM call when data fully speaks for itself. (Subject to "never empty" rule — planner decides.) |

### 4.4 Routing Integration

| # | Decision | Rationale |
|---|---|---|
| 22 | **Planner does NOT replace `unknown`.** It slots in as a *deferred* target for confidence-gated stages. | Naive "replace unknown" leaves two confidently-wrong upstream stages (LLM classifier, `matchQuery`) grabbing adjacent queries first. |
| 23 | **LLM classifier returns `{ capability, confidence: 'high' \| 'medium' \| 'low' }`** (string buckets, not numeric — LLMs aren't calibrated probabilists). `high` → wired cap. `medium` or `low` → planner. Tunable later. | Gives the classifier a "defer" output it lacks today. |
| 24 | **`matchQuery` returns confidence**, downgraded by dimensional signals. Strong match → wired uc1/2/3. Weak/dimensional match → planner. | Today `matchQuery` over-fires on uc1 for any "magnolia" query, even comparative/temporal ones it can't actually answer. |
| 25 | **Dimensional downgrade signals for `matchQuery`:** time qualifiers (`last month`, `this quarter`, `vs`, `compared to`, `since`, `between`); group/sort qualifiers (`by rep`, `by collection`, `grouped`, `top N`, `sorted`); multi-entity (>1 customer/lead/rep named); non-canonical entities (name not the fixture's keyed entity). | Aligned. |
| 26 | **SKU-lookup guard hands off to planner with a hint** (`{ kind: 'unknown_sku', extracted: '...' }`) instead of short-circuiting to `unknown`. | Planner can query products or render a clean "no SKU matched" — either way better than a hard stop. |
| 27 | **Kill switch:** `KAI_PLANNER_DISABLED` env var / settings flag. Disabled → planner stage skipped, falls through to fallback card. | Cheap insurance for demo day. |
| 28 | **Hybrid mode (planner augmenting wired cap)** — punted entirely for v1. No reserved hooks (avoids dead code). | Cleaner addition in v2 when shape is known. |
| 29 | **Conversation history visible to planner** — deferred to v2. v1 planner is pure single-turn. | Easier to debug + evaluate. |

### 4.5 Observability & Infra

| # | Decision | Rationale |
|---|---|---|
| 30 | **Models:** Opus 4.6 for planner. Sonnet 4.6 for closing text + widget data hydration. | Plan generation benefits from Opus reasoning; Sonnet is fast/cheap enough for closing text. |
| 31 | **Logging (v1):** structured console + `window.lastPlannerRun` debug payload. | Quick inspection; persistence is v2. |
| 32 | **Persistent planner logging** (database + Streamlit dashboard) — v2. | Not needed for v1 evaluation. |
| 33 | **Chips below planner responses:** generic ("Save to artifacts", "Show me more") for v1. Closest wired-cap chips by entities touched if easy in v1.5. Full dynamic chips in v2. | Avoids dead-end feel without overbuilding. |
| 34 | **Widget catalog + planner machinery live in new** `src/lib/planner/` **directory.** | Self-contained, easy to iterate or delete. |
| 35 | **Decision.md lives at** `plan_docs/Decision.md`. | `plan_docs/` is the planning home. |

## 5. The Plan IR Schema (locked)

```ts
type Plan = {
  reasoning: string;                     // shown live in UW-014, then collapsed
  steps: Array<{
    tool: string;                        // one of the 6 wide query tools
    args: object;                        // tool-specific filter object
    bindTo: string;                      // identifier for downstream refs
  }>;
  widgets: Array<{
    widgetType: string;                  // must exist in widget catalog
    dataFrom: string;                    // refs a step's bindTo
    transform?: {                        // optional shape adjustment
      pick?: string[];
      rename?: Record<string, string>;
      // more as needed
    };
    config?: object;
    highlights?: WidgetHighlight[];
  }>;
  closingTextHint: string | null;        // seed for Sonnet's closing pass; null = skip
  emptyResultInsight?: boolean;          // true → closing text uses insight-style, not summary-style
};
```

Compilation: IR → run `steps` in order → hydrate each widget's `data` from `dataFrom` step result + transform → produce Frame JSON → existing FrameParser renders.

## 6. Updated Query Routing Order

```
1. Page Context              [fast-path, unchanged]
2. Dashboard Routes          [fast-path, unchanged]
3. Special Routes            [fast-path, unchanged]
4. Chip Spawns               [fast-path, unchanged]
5. Capability Routes         [fast-path, unchanged]
6. Email Pre-router          [fast-path, unchanged]
7. SKU-lookup guard          → on miss, HAND OFF to planner with hint
                              (was: terminate to `unknown`)
8. LLM classifier            → returns { capability, confidence }
                              high → wired cap
                              medium/low → PLANNER
9. matchQuery (general)      → returns { match, confidence }
                              strong → wired uc1/2/3
                              weak/dimensional → PLANNER
10. PLANNER                  → IR → tool calls → Frame JSON
                              on failure → fallback card
11. Fallback card            → honest copy + 2-3 wired-cap chips
```

## 8. Build Order (v1)

1. **Data tool layer** — `src/lib/planner/tools/` — 6 typed query functions over `AUDREY_*` arrays with filter args, embedded aggregations, auto-hydration, 50-row cap.
2. **Widget catalog** — `src/lib/planner/widget_catalog.ts` — one entry per registered widget.
3. **Plan IR types + zod schema** — `src/lib/planner/types.ts`, `src/lib/planner/schema.ts`.
4. **Planner route** — `src/app/api/kai/plan/route.ts` — Opus 4.6, system prompt assembled from tool catalog + widget catalog + (eventually) exemplars from the ground-truth set.
5. **IR compiler** — `src/lib/planner/compile.ts` — IR → Frame JSON, validates per-widget data shape.
6. **Routing integration** — modify `ChatShell.handleSend` + `queryMatcher.ts` per §6: confidence gating on classifier and `matchQuery`, SKU-guard handoff, planner terminal.
7. **UW-014 streaming behavior** — live step trail → collapse summary.
8. **Closing text sequencing** — wait for all widget hydration before streaming.
9. **Fallback card** — copy + chip selection from `action-chips-map.json` by touched entities.
10. **Kill switch + logging** — env var, `window.lastPlannerRun`, structured console.
11. **Evaluation** — run against ground-truth set when received. Tune thresholds + tool filters.
12. **`npx tsc --noEmit`** must pass at every step (per `CLAUDE.md` Rule 1).
